## 定位

P7：**使用者畫像 / DMP（資料管理平台）**——把 GA4 真使用者資料做成 **RFM/LTV/行為標籤體系 → 事件級互動式 OLAP → 人群圈選（crowd-selection）→ 管理後台**。打 **DE / 資料分析 JD**。這是「不只算模型，還能把使用者資料做成可被行銷操作的資產」的能力展示。

**上游合約已鎖**（地基 design）：`gold_ga4_user_rfm`（user 級 RFM/LTV 度量**源**，非分數——分群切點是 P7 決策）；`gold_ga4_sessions`（**只含漏斗活躍 session 非全 session**，session 行為標籤源）；`gold_ga4_item_catalog`（item 側標籤）；`silver.ga4_events`（事件級明細，ClickHouse OLAP 的鏡像源）。

## 已鎖定決策（勿翻案）

1. **資料源只用 GA4 地基產出的合約**（公開 sample，area02 零進入）。
2. **＋ClickHouse 翻案落地於此**（NORTH_STAR 已批准）：ClickHouse ＝**GA4 事件流互動式 OLAP**（人群圈選 DSL 的秒級查詢後端；切片/漏斗/留存）。**用的是 ClickHouse 的欄式 OLAP 正典用途，非十億級 bitmap 圈人**——我方規模 bitmap 用不上，**design 的 ADR 段必須誠實寫明這點**（NORTH_STAR 三工具翻案表已定調）。ClickHouse 是本垂直**唯一**新增元件、有獨特職務（Postgres 是 OLTP+Gold 聚合、非欄式 OLAP 引擎）。
3. **標籤/分群體系是 P7 造，地基只出度量**（`gold_ga4_user_rfm` 是「源」不是「分數」）：RFM 分箱切點、價值分層、行為標籤規則在本 spec 定義。
4. **人群圈選 DSL**：定義一套**標籤條件 DSL**（match/range/exists 類條件 → 編譯成 ClickHouse 查詢），非硬編碼查詢。取材課程「通用標籤條件 DSL（→ ES bool query）」的**邏輯**，載體換成 ClickHouse SQL（我方無 ES，一工一具不新增）。
5. **前端仍 Vercel 靜態，ClickHouse 是叢集內服務**：P7 對 P4 的合約＝「**批次產生的畫像/分群摘要表 → P4 匯出 DAG → 靜態 JSON**」（分群佔比、標籤覆蓋、RFM 熱圖等前端可展示）；**ClickHouse 互動式圈選 + admin 是叢集內能力**，以 **MCP 工具 + 截圖/GIF + 負載測試**佐證，非公開靜態站 runtime 依賴（守拓撲鐵律）。
6. **admin 後台右尺寸**：圈選/標籤管理 admin 是**叢集內服務**（非 Vercel 前端）。取材課程「amis 低代碼 admin」的**概念**，但**不引入 amis**（一工一具/成本）——用最輕載體（design 收斂：簡單 Next.js/RSC admin 頁 or FastAPI+輕前端，跑叢集內），重點是**圈選 DSL + 標籤 CRUD + 人群預覽**功能，不是低代碼框架本身。
7. **分群模型守 M4 + P2 慣例**：若用 KMeans/XGBoost 做價值分層，走 M4 host 訓練 + MLflow 追蹤（沿 P2 tabular 慣例）；但**優先規則式分箱**（RFM 傳統五分位規則簡單可解釋、無需模型）——design 收斂規則式為主、模型為輔（進化方向）。
8. **一工一具仍守**：排程只 Airflow、離線 DB Postgres、OLAP 只 ClickHouse（新增、有獨特職務）、agent 只 LangGraph、微調只 HuggingFace。不引 ES/amis/第二個 OLAP。
9. **前端說明式 UI（硬性，跨 P4/P6/P7）**：畫像/分群前端頁帶 ga-insight 式三層說明（InfoTooltip/ChartCaption/Explainer，定義類預設展開·方法論類收合）——**這正是 ga-insight `05_客戶價值分群.py` 的主場**（RFM 8 種行為分群 + 4 級價值，`expanded=True` 定義表），直接對照取材。

## 範圍

1. **標籤體系（tag schema + 規則）**：
   - **RFM 分層**：`gold_ga4_user_rfm` 的 R/F/M 度量 → 五分位分箱 → RFM 分數 → 8 種行為分群（冠軍/忠誠/潛力/新客/流失風險/沉睡…，對照 ga-insight）+ 4 級價值等級。切點與命名定案。
   - **行為標籤**：從 `gold_ga4_sessions`（漏斗活躍 session）衍生——瀏覽深度、加購未購、類別偏好（接 `gold_ga4_item_catalog` 的 item_category）等。
   - **標籤時間衰減權重**（取材課程「tag time-decay」）：近期行為權重高，design 定衰減函式（但守誠實：靜態資料集錨定 `data_anchor_date` 非 now()，同地基 RFM 處理）。
   - **標籤落點**：Postgres 標籤表（user_pseudo_id → 標籤集）+ 鏡像進 ClickHouse 供圈選。schema 欄位級。
2. **ClickHouse 事件 OLAP 層**：
   - **建表**：`silver.ga4_events`（+ 標籤表）鏡像進 ClickHouse（MergeTree 家族，design 定 engine/排序鍵/分區）。同步機制（Airflow task：Postgres/Iceberg → ClickHouse，右尺寸——非 CDC，批次載入）。
   - **互動式查詢能力**：漏斗、留存、標籤交叉、人群大小估算——展示 ClickHouse 欄式 OLAP 秒回（對比 Postgres 掃事件會慢的 ADR 論證）。
3. **人群圈選 DSL**：
   - **DSL 定義**：標籤條件（match/range/exists/and-or 組合）的結構化表示（JSON schema 欄位級）。
   - **編譯器**：DSL → ClickHouse SQL（取材課程 DSL→ES bool query 的**邏輯**，載體換 ClickHouse）。
   - **人群預覽**：給 DSL → 回人群大小 + 樣本 + 標籤分佈。
4. **admin 後台（叢集內）**：標籤 CRUD、圈選 DSL 建構器、人群預覽、標籤覆蓋儀表。含 ExplainerSection（自我說明）。
5. **前端展示頁（Vercel 靜態，走匯出合約）**：分群佔比、RFM 熱圖、標籤覆蓋、價值分層——批次匯出 JSON，帶三層說明式 UI（對照 ga-insight 分群頁）。
6. **（可選/進化方向）價值分層模型**：KMeans/XGBoost 分群（沿 P2 MLflow），design 判斷 v1 是否納入或列進化方向（傾向規則式 v1、模型列進化）。

## 開放問題（design 收斂，禁 TBD，皆附傾向）

1. **ClickHouse 部署形狀**：k8s 內單機 vs 叢集。傾向**單機 Deployment + PVC**（portfolio 規模，叢集是過度工程，ADR 誠實註記「示範欄式 OLAP 能力非叢集運維」）。
2. **ClickHouse 同步機制**：CDC vs 批次載入。傾向**Airflow 批次載入**（右尺寸，靜態資料集無需 CDC；一工一具不引 Debezium/Kafka-connect）——design 定同步 DAG 形狀。
3. **RFM 分箱**：五分位 vs 自訂切點。傾向**五分位**（傳統、可解釋、對照 ga-insight）；平手/邊界處理定案。
4. **行為分群數與命名**：傾向對照 ga-insight 8 種行為分群 + 4 級價值（招募方認得的標準體系）——design 定對照表。
5. **DSL 表達力**：支援到哪（match/range/exists/and-or 夠不夠）。傾向**match/range/exists + and/or 巢狀**（涵蓋 90% 圈選需求，不做圖靈完備 DSL）——design 定 JSON schema。
6. **標籤時間衰減**：衰減函式（指數 vs 線性 vs 分段）。傾向**指數衰減 + 半衰期參數**，錨定 `data_anchor_date`——design 定。
7. **admin 載體**：Next.js RSC（同前端技術棧）vs FastAPI+輕前端。傾向 **Next.js admin（叢集內，非 Vercel 那份）**（技術棧一致、ExplainerSection 可複用概念）——design 定並說明與 Vercel 靜態前端的區別。
8. **價值分層 v1**：規則式 vs KMeans。傾向**規則式 v1**（RFM 分箱即分層，簡單可解釋）、KMeans/XGBoost 列進化方向。

## 共用契約（跨 spec 接縫，design 必須明訂）

- **接縫 E｜sessions 是漏斗活躍 session 非全 session**（地基 §5.3 明訂）：行為標籤衍生不得假設全 session 覆蓋；純瀏覽 session 不在源內（如需，走地基 additive 新表，見地基 §13 known-limit 3）。
- **接縫 F｜RFM 是源非分數**（地基 §5.4）：分箱/分層在本 spec，`recency_days` 已錨 `data_anchor_date`。
- **接縫 G｜item_category 標籤**（地基 §5.2）：類別偏好標籤接 item_catalog。
- **接縫 H｜P7→P4 匯出**：只加檔/加欄（P4 §4 additive）。
- **接縫 I｜與 P6 的關係**：P7 標籤**可**餵 P6 推薦（分群推薦/冷啟），但 v1 **不強耦合**——design 定「標籤表可被 P6 讀取」的鬆接縫，不做同步依賴。

## 設計約束（硬性）

- 精確度契約 8 條自檢。
- 一工一具（ClickHouse 唯一新元件、有獨特職務、ADR 誠實寫明非 bitmap 規模）；不引 ES/amis；離線仍 Postgres。
- 拓撲：ClickHouse+admin 叢集內、前端 Vercel 靜態走匯出合約打不到 k8s；additive（不改地基合約、不改 P4 既有匯出）。
- M4 界線（任何模型訓練走 host）；靜態資料集誠實（時間衰減/recency 錨 `data_anchor_date`）。
- 進化非複刻：取材課程（RFM/分群/標籤衰減/圈選 DSL/amis admin）標「取邏輯 vs 重造/替換工程層」——DSL→ES 改 DSL→ClickHouse、amis→輕 admin。
- 隱私誠實：`user_pseudo_id` 裝置級匿名（地基 §13），畫像是「裝置畫像」非「真人身分」，如實敘述。

## 交付與驗收

- 標籤體系（RFM 分箱 + 行為分群 + 行為標籤 + 時間衰減）schema 欄位級 + 規則定義。
- ClickHouse 建表（engine/排序鍵/分區）+ 同步 DAG 形狀。
- 圈選 DSL（JSON schema 欄位級）+ DSL→ClickHouse SQL 編譯器 + 人群預覽。
- admin 後台（標籤 CRUD/圈選建構器/人群預覽/ExplainerSection）。
- 前端展示頁（分群/RFM 熱圖/標籤覆蓋批次 JSON + 三層說明式 UI）。
- ClickHouse OLAP 能力佐證（MCP/截圖/負載測試：欄式秒回 vs Postgres 對照）。
- 端到端可跑驗收：GA4 Gold → 標籤 → ClickHouse → 一條 DSL 圈選回人群大小+樣本 → 前端分群 JSON 可查。
- plan-前實查點清單（帶預設傾向）。
